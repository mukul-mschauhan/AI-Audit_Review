Build an enterprise-grade AI-powered Audit Evidence Review Assistant
designed to be demonstrated to Big 4 audit leadership. The purpose
of this application is to make Agentic AI observability, traceability,
and ROI visible to non-technical business leaders.

This is NOT a generic dashboard. The observability layer is the
primary differentiator. Every LLM call must be instrumented. Every
agent step must be traceable. Every dollar of token spend must be
visible. Every audit finding must be defensible with a reasoning log.

====================================================================
TECH STACK
====================================================================
- Frontend: React (Vite) + Tailwind CSS + shadcn/ui + Recharts
- Backend: Node.js + Express
- Database: PostgreSQL with Drizzle ORM (Replit default)
- AI: OpenAI API directly (model: gpt-4o; fallback: gpt-4o-mini)
- File parsing: pdf-parse, mammoth (docx), xlsx (sheetjs), sharp (images)
- State: TanStack Query for server state
- Routing: wouter

AGENT ARCHITECTURE (this is mandatory — do not collapse into one LLM call)
====================================================================
Implement a 4-agent pipeline. Each agent is a separate LLM call with
its own system prompt, its own model, its own span in the trace.

Agent 1 — EXTRACTOR
  Input: raw text from uploaded document
  Job: extract structured entities (vendor, invoice_number, amount,
       currency, date, PO_reference, approver_name, contract_id,
       payment_terms)
  Output: JSON

Agent 2 — CONTROL VALIDATOR
  Input: extracted entities + audit control rules
  Job: check each of the 5 controls (missing approval, missing
       signature, threshold breach, expired contract, duplicate
       invoice) and return violation list with evidence
  Output: JSON with control_id, violated (bool), evidence_quote, confidence

Agent 3 — RISK CLASSIFIER & FINDING DRAFTER
  Input: violations from Agent 2
  Job: classify each violation as Low/Medium/High/Critical and draft
       an audit observation (finding, impact, recommendation) in the
       house style of a Big 4 audit report
  Output: JSON list of findings

Agent 4 — REFLECTION / CRITIC (this is the Module 2 Reflection pattern)
  Input: findings from Agent 3 + original extracted entities
  Job: critique the draft findings. Flag any finding that is
       (a) not supported by evidence in the document,
       (b) over-classified relative to the evidence,
       (c) missing a recommendation.
       Return the revised, approved findings.
  Output: JSON list of approved findings + critic_notes

All four agents run sequentially per document. Each agent's input
and output is persisted. Each agent's call to OpenAI is wrapped in
an instrumentation layer (see next section).

====================================================================
OBSERVABILITY LAYER (this is the demo's core)
====================================================================
Create a TRACE table in Postgres with the following columns:
  trace_id (uuid)          — one per document
  span_id (uuid)           — one per LLM call or tool call
  parent_span_id (uuid)    — nullable, for the trace tree
  document_id (fk)
  agent_name (text)        — extractor, validator, classifier, critic
  span_type (text)         — llm_call, tool_call, parse, validation
  model (text)             — gpt-4o, gpt-4o-mini
  input_tokens (int)
  output_tokens (int)
  total_tokens (int)
  cost_usd (numeric(10,6))
  latency_ms (int)
  status (text)            — success, error, timeout, rate_limited
  error_message (text, nullable)
  started_at (timestamp)
  ended_at (timestamp)
  request_payload (jsonb)   — full prompt sent
  response_payload (jsonb)  — full response received
  reasoning_summary (text)  — 1-line human-readable summary of what
                              this span decided and why

Every OpenAI call must go through a single wrapper function
`callOpenAI(agentName, model, messages, parentSpanId, traceId)` that:
  1. Records start time
  2. Calls OpenAI
  3. Reads usage.prompt_tokens, usage.completion_tokens from response
  4. Computes cost from a CONFIGURABLE pricing table:
       gpt-4o:        $2.50 input / $10.00 output per 1M tokens
       gpt-4o-mini:   $0.15 input / $0.60 output per 1M tokens
     (store as config so rates can be updated)
  5. Inserts a row into the TRACE table
  6. Returns the response
  7. On error: still inserts a row with status='error' and message

Also instrument NON-LLM steps:
  - PDF parsing → span_type='parse'
  - Control rule check (deterministic) → span_type='validation'
  - Database read/write → span_type='tool_call'

====================================================================
PREDEFINED CONTROL RULES (deterministic, not LLM-based)
====================================================================
The Validator agent uses the LLM to extract intent but the final
control violation is computed deterministically (this is the
Module 1 lesson: precision-critical work uses rules, not GenAI).

Hardcode these 5 rules:
  R1 Missing approval:    invoice amount > 50,000 AND approver is null
  R2 Missing signature:   document type = invoice/contract AND signature_present = false
  R3 Threshold breach:    invoice amount > vendor_contract_max_value
  R4 Expired contract:    invoice date > contract_end_date
  R5 Duplicate invoice:   invoice_number already exists in DB for same vendor

Each rule has a fixed default risk weight (configurable in admin UI).
