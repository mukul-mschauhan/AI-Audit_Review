# AI Audit Review

AI Audit Review is a professional multi-agent audit intelligence application that reviews contracts, invoices, bills, legal documents, and supporting evidence to identify anomalies, control violations, duplicate risks, missing approvals, and compliance gaps.

The application is designed as an enterprise-grade audit review demonstrator. It does not only generate findings; it also shows how those findings were produced through traceability, observability, governance controls, evaluation metrics, and human review workflows.

## Why This Project Exists

Traditional audit review is often slow, sample-based, expensive, and difficult to trace end-to-end. Manual reviewers may examine only a small portion of the available document universe, which creates risk around missed anomalies, duplicate payments, incomplete evidence trails, and inconsistent reviewer judgement.

AI Audit Review demonstrates how an AI-powered review system can:

- Review documents at scale
- Extract structured information from unstructured evidence
- Detect anomalies and control violations
- Classify findings by severity
- Recommend corrective actions
- Show evidence behind every finding
- Maintain audit logs and traceability
- Escalate uncertain outputs to human reviewers
- Improve system quality through feedback and evaluation

## Core Concept

The application uses a sequential multi-agent audit workflow.

```text
Document Upload
      ↓
Extractor Agent
      ↓
Validator Agent
      ↓
Classifier Agent
      ↓
Critic / Reflection Agent
      ↓
Audit Finding + Evidence + Recommendation
```

Each agent performs a specific role and produces traceable intermediate outputs. This makes the system easier to inspect, govern, and explain than a single black-box LLM response.

---

## Multi-Agent Architecture

### 1. Extractor Agent

The Extractor Agent reads the source document and converts unstructured content into structured audit fields such as:

- Vendor name
- Invoice or contract amount
- Dates
- Approval status
- Contract reference
- Currency
- Payment terms
- Authorization chain
- Supporting evidence

Its job is to create a structured representation of the document so that downstream agents can reason over clean data.

---

### 2. Validator Agent

The Validator Agent applies audit control checks against the extracted data.

Example control checks include:

- Dual approval requirement
- Amount threshold validation
- Vendor verification
- Contract expiry or currency mismatch
- Duplicate invoice or payment detection
- Missing supporting documentation
- Unauthorized approval path

Every violation must be linked to source evidence. The system is designed so that findings should not be accepted unless they can be grounded in document text or extracted evidence.

---

### 3. Classifier Agent

The Classifier Agent converts validated issues into formal audit findings.

It classifies each issue by:

- Risk severity: Critical, High, Medium, Low
- Business impact
- Financial exposure
- Compliance impact
- Recommended corrective action
- Confidence score

This agent turns raw anomalies into audit-ready findings.

---

### 4. Critic / Reflection Agent

The Critic Agent independently reviews the findings generated by the Classifier Agent.

Its role is to challenge the output before it reaches the auditor or business user.

The Critic Agent checks:

- Is the finding supported by evidence?
- Is the risk severity correct?
- Is the recommendation practical?
- Is the confidence score reasonable?
- Should the finding be escalated for human review?

This reflection pattern helps reduce hallucination, overstatement, weak evidence, and misclassification.

## Key Product Features

### 1. Executive Overview

A leadership-level dashboard that summarizes the audit review status.

It shows:

- Total documents analyzed
- Total findings generated
- Critical findings count
- Compliance rate
- Token usage
- AI processing cost
- Risk distribution
- Cost-per-document trend
- ROI snapshot

This tab is meant for senior stakeholders who want to quickly understand risk exposure, processing efficiency, and business impact.

---

### 2. Upload & Analyze

The document processing workspace.

Users can upload documents or run a pre-seeded demo sequence. The platform shows a live trace of the agent pipeline as each document moves through the audit review workflow.

The live trace includes:

- Active agent name
- Model used
- Token count
- Cost
- Latency
- Processing status

This helps demonstrate that the system is not a black box.

---

### 3. Trace Explorer

The Trace Explorer provides deep visibility into how each finding was produced.

It allows users to inspect:

- Agent-by-agent execution timeline
- Span-level trace details
- Prompt and response flow
- Token usage
- Cost attribution
- Latency
- Source evidence
- Reasoning summary
- Model output

This is the auditability layer of the application.

A reviewer can trace a final audit finding back to the original document evidence and the exact agent steps that produced it.

---

### 4. Cost & Performance Analytics

This section tracks the operational performance of the AI audit pipeline.

It includes:

- Spend trend
- Cost per document
- Token consumption by agent
- Latency percentiles
- Model usage mix
- Cost per finding
- Cost by risk tier

This is useful for CFOs, audit transformation teams, and AI governance teams that need to understand whether the system is economically viable at scale.

---

### 5. Governance

The Governance module shows the control layer around the AI system.

It includes:

- Human-in-the-loop review queue
- Low-confidence finding escalation
- Failure mode tracking
- Audit log
- User-attributed actions
- Approval and rejection history

The goal is to show that AI does not operate without oversight. Findings that are uncertain, unsupported, or high-risk can be routed to human reviewers before final acceptance.

---

### 6. Observability & Evaluation

The Observability module measures AI quality and system reliability.

It tracks:

- Agent performance
- Override rate
- Groundedness
- Confidence distribution
- Hallucination checks
- Escalation rate
- Parse failure rate
- Evaluation scorecards
- Model comparison

This module helps answer an important enterprise question:

> How do we know the AI system is reliable?

Instead of assuming quality, the application makes quality measurable.

### 7. Business Case

The Business Case module translates technical performance into financial value.

It compares AI-assisted audit review against manual review across:

- Time per document
- Cost per document
- Coverage
- Reviewer consistency
- Scalability
- Annual savings
- Payback period
- Net present value

This section is designed for decision-makers who need a business justification, not only a technical explanation.

### 8. Executive Report

It can include:

## AI Safety and Trust Design

The project is designed around trust, traceability, and audit-readiness.

Key safeguards include:

### Evidence Grounding

Every audit finding should be connected to source evidence. Unsupported findings should be rejected or escalated.

### Critic Agent Review

The Critic Agent independently challenges the output of the Classifier Agent before the finding is accepted.

### Confidence-Based Escalation

Low-confidence findings are routed to the human-in-the-loop queue.

### Immutable Audit Trail

Every major action is logged with timestamp and attribution.

### Full Span Traceability

Each finding can be traced through the full agent chain:

```text
Extractor Span → Validator Span → Classifier Span → Critic Span → Final Finding
```

## Installation

Clone the repository:

```bash
git clone https://github.com/mukul-mschauhan/AI-Audit_Review.git
cd AI-Audit_Review
```

Install dependencies:

```bash
pnpm install
```

Run type checks:

```bash
pnpm run typecheck
```

Build the workspace:

```bash
pnpm run build
```

Use the following flow when presenting the application.

Explain the business problem:

- Manual review is slow.
- Sampling misses exceptions.
- Audit traceability is often weak.
- AI can improve coverage, speed, and consistency.

```text
Extractor → Validator → Classifier → Critic
```

- Groundedness

It should not be used as the sole basis for audit opinions, legal conclusions, regulatory filings, or financial decisions without qualified human review.

The system is designed to augment auditors, reviewers, and compliance teams by improving speed, coverage, traceability, and evidence organization.
