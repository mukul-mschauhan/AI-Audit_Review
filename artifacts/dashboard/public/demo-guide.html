2 · Problem Statement — Why This Exists

The Audit Evidence Review Challenge

In a modern Big 4 audit engagement, the volume of supporting documents — invoices, contracts, bank statements, authorization letters — far exceeds the capacity of human reviewers to examine individually. Teams rely on statistical sampling: reviewing only a fraction of documents and inferring compliance from that sample.

Current State — The Gaps

  • Only 8% of documents are reviewed. 92% are never examined — including the documents that contain the most significant findings.
  • $187.50 per document. At $250/hr and 45 minutes per document, costs scale linearly with volume.
  • Reviewer variance of ±18%. Different reviewers classify the same document differently. Findings are inconsistent.
  • No traceability. There is no documented chain between a finding and the specific text in the source document that supports it.
  • Duplicate payments slip through. Sampling cannot reliably detect cross-document patterns like duplicate invoices from different vendors.
  • Manual process does not scale. More documents require proportional headcount. There is no efficiency gain.

Consequences for the Firm

  • Regulatory exposure. Undetected control violations create liability. Auditors sign opinions on incomplete evidence.
  • Financial leakage. Average duplicate payment incident: $124K. Sampling cannot catch what it does not see.
  • Indefensible opinions. Without a complete audit trail, findings cannot be substantiated in regulatory review or litigation.
  • Competitive disadvantage. Firms adopting AI-first audit processes will offer more thorough coverage at lower cost, capturing clients.
  • Talent misallocation. Senior auditors spend time on mechanical extraction that adds no judgment value.

The Fundamental Tension

Thoroughness and cost are in direct conflict under the current model. The only way to review more documents is to spend more hours. This platform eliminates that trade-off.

3 · Solution Architecture — The 4-Agent Pipeline

The platform uses a sequential multi-agent architecture. Each agent is a discrete, auditable LLM call with its own system prompt, model assignment, and trace span. Precision-critical control checks use deterministic rules; LLMs handle language-intensive tasks (extraction, classification, reflection).

1

Extractor

gpt-4o · Span: sp-extract-*

Task: Parse the raw document (PDF/DOCX/XLSX) and extract structured entities into validated JSON — vendor, amount, date, contract reference, approval status, authorisation chain.

Why gpt-4o: Document parsing requires strong reading comprehension across diverse formats. Accuracy here is foundational — downstream agents trust this output.

Observable: Full system prompt, user message (doc text), output JSON, and token split all visible in Trace Explorer.

2

Validator

gpt-4o · Span: sp-validate-*

Task: Apply 5 deterministic control rules (R1–R5) against the extracted data. For every violation, the Validator must cite an evidence_quote — the exact text from the source document that supports the finding.

Control rules: R1: Dual approval · R2: Amount threshold · R3: Vendor verification · R4: Contract currency & expiry · R5: Duplicate detection.

Why this matters: The evidence_quote is the foundation of groundedness. If the AI cannot cite text, the finding is rejected.

3

Classifier

gpt-4o · Span: sp-classify-*

Task: For each validated violation, classify risk as Critical / High / Medium / Low. Draft the formal audit finding, business impact, and recommended management action.

Output structure: Every finding includes risk, finding, impact, recommendation, and a confidence score.

Confidence gate: Findings below 70% confidence are automatically flagged for human-in-the-loop review before they reach the output.

4

Critic (Reflection Agent)

gpt-4o · Span: sp-critic-*

Task: Independently review every finding produced by the Classifier. The Critic does not see the Classifier's reasoning — it reads the original evidence and forms its own assessment. If it disagrees, it overrides the risk classification and documents why.

Pattern: This is the Reflection agentic pattern — the AI checks itself. It is the primary defence against hallucination and misclassification.

Override rate: ~14% of Classifier outputs are revised by Critic. This is healthy — it means the system is catching errors before they reach the auditor.

4 · AI Safety & Trust Framework

Audit is a high-stakes, regulated domain. This platform is designed to be trusted, not just used. Every safeguard is observable and demonstrable in the dashboard.

The 5 Safety Mechanisms

① Groundedness Check

Every finding from the Validator must include an evidence_quote — exact text from the source document that supports the finding. If no quote exists, the finding is not produced. This makes hallucination structurally impossible at the Validation stage.

Metric: 100% groundedness across all findings in the current dataset.

② Reflection / Critic Agent

Agent 4 independently reviews all Classifier outputs. It has no memory of Agent 3's reasoning — it approaches the evidence fresh. This mirrors the principle of independent reviewer sign-off in traditional audit.

Metric: ~14% Critic override rate. Every override is logged and visible.

③ Confidence-Gated Escalation

Every finding carries a confidence score. Findings below the 70% threshold are automatically placed in the Human-in-the-Loop queue and cannot proceed to the final output without human approval.

Tab: Governance → Human-in-the-Loop Queue

④ Immutable Audit Log

Every action — document upload, agent call, finding generated, human approval/rejection — is written to an append-only audit log with a timestamp and user attribution. This log cannot be edited and is available for regulatory inspection at any time.

Tab: Governance → Audit Log

⑤ Full Span Traceability

Every approved finding carries the IDs of all spans that produced it — Extractor span → Validator span → Classifier span → Critic span. Regulators can request a complete trace for any finding and receive the full chain: the document, the prompts, the model outputs, and the confidence scores at each step. This is the "explainability" story for regulatory conversations.

8 · Common Questions & Answers

Questions likely to be raised by audit leadership, technology committees, and risk officers — with suggested responses.

QuestionSuggested Answer
"What happens when the AI gets it wrong?" We expect the AI to get things wrong occasionally — that's why we designed for it. Agent 4 (Critic) independently reviews all findings and overrides about 14% of them. Anything below 70% confidence goes to a human reviewer before it reaches output. And every action is logged. The question isn't "is the AI perfect?" — it's "is this process safer and more complete than manual sampling?" The answer is yes on both counts.
"Is our client data secure?" Documents are processed in isolated compute environments. There is no training on client data — OpenAI's enterprise API contract explicitly prohibits this. Every access is timestamped and attributed in the audit log. We can configure on-premise or private cloud deployment for engagements with heightened data sensitivity.
"Can we trust findings from an AI for a signed audit opinion?" The AI doesn't sign the opinion — the partner does. The platform surfaces findings with evidence, confidence scores, and full traceability. The audit team reviews and approves each finding. The AI dramatically increases the number of documents reviewed and the consistency of that review, but every finding is human-approved before it reaches the report.
"What if the AI model changes or degrades?" The Eval Suite (Observability tab) runs a 7-dimension benchmark comparison on demand. We can detect quality degradation before it affects outputs. The model is also abstracted — we can swap providers by changing a single configuration value, without touching any application code.
"How does this integrate with our existing audit tools?" The platform outputs structured JSON per finding — compatible with standard audit management platforms. The Export PDF function generates reports that match existing working paper formats. API integration with document management systems (SharePoint, iManage) is a standard integration layer we can build in Phase 2.
"What about regulatory acceptance of AI-assisted audit?" The platform is designed for the regulatory conversations that are coming. Full traceability means every finding can be supported with a complete chain: document → agent call → evidence quote → finding. PCAOB and FRC guidance increasingly accepts AI-assisted audit where the AI's reasoning is documented and human judgment is applied to outcomes. This platform provides exactly that documentation.
"What's the implementation timeline?" The core platform is production-ready as shown. A firm-specific deployment — custom control rules, document template calibration, integration with existing DMS — typically takes 4–6 weeks. A pilot engagement on one client portfolio can begin within 2 weeks of commercial agreement.
"Could we have built this ourselves?" You could — the technology is accessible. The IP here is in the design choices: the agent architecture, the control rule library, the safety framework (groundedness, reflection, escalation), and the observability instrumentation. Building and validating this in-house would typically take 12–18 months and $800K–$1.2M in engineering cost. We're offering it ready to deploy.
